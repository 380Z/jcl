JEDI Issue Tracker

Quick start guide



Getting started


First you need to signup for a new account. The issue trackers login screen provides a link (Figure 1).





The following sign up screen will ask you for a username and a valid email address. A randomly generated password will be send to that email address shortly afterwards. Once you have your randomly generated password please go back to the login page.


Just enter your username and password and hit the login button. There is also a Save Login checkbox to have the package remember that you are logged in between browser sessions. You will have to have cookies enabled to login.


If the account doesn't exist, the account is disabled, or the password is incorrect then you will remain at the login page. An error message will be displayed.



The main screen


The main screen is the first page you see upon logging in. It shows you the latest news and updates for the bugtracker. Some news postings are specific to projects and others are global across the entire bugtracker. This is set at the time of posting in the section. In the top right corner you are able to switch between the different projects. It is recommended to set it to Code Library now.


When you first logged in you have the following options:





Main

The screen you are seeing now. You can always get back to it by pressing that button.

View Bugs

Shows all bugs reported in the selected project.

Report Bug

Report a bug

Summary

Some statistics about bugs, longest open bugs etc.

Account

You account settings

Docs

Documentation related to the bucktracker or the code library.

Logout

Exit the bug tracker


The first thing you as developer should do now is to write a short message to jcl@delphi-jedi.org and ask for developer rights. The next step is most likely to customize your account.





Now you are able to change and confirm your password. Please just enter it into the appropriate fields. Afterwards please go to the preferences page. There you can turn on/turn off every kind of email notification. At startup you will not get any email notifications at all.

Email you when a bug you are attached to is set to assigned

Email you when a bug you are attached to is set to Feedback status

Email you when a bug you are attached to is set to Resolved status

Email you when a bug you are attached to is set to Closed status

mail you when the status of a bug you are attached to has changed

Every email you get will have the sender address bugtracker@delphi-jedi.org - you can use it as sort criteria within you email application.



View bugs


The bugs are listed in a table and the attributes are listed in the following order: priority, id, number of bugnotes, category, severity, status, last updated, and summary.





Each (except for number of bugnotes) can be clicked on to sort by that column. Clicking again will reverse the direction of the sort. The default is to sort by id in descending order.

The bug id is a link that leads to a more detailed report about the bug. Depending on what you have set in your Account Preferences you will be sent to the simple or advanced view. You can also add Bugnotes here.

Each table row is color coded according to the bug status. Here is an explanation of the default colorings:

red

Bug is new

purple

Feedback - bug requires more feedback before work can proceed

organge

Acknowledged - lets the user know that the bug has been examined but probably not by the proper developer

yellow

Confirmed - bug has been confirmed by an updater or developer

blue

Assigned – A developer is working on the bug.

green

Resolved - bug should be fixed, waiting on confirmation of fix

gray

Closed – Bug is closed



Used bugstates


These bug states introduce some kind of overhead for a project like JEDI. Therefore we normally use only the new, assigned, Resolved and closed bug states. Each bug is new by default. If you want to work on it please assign yourself to it (bugstate assigned). When you are done simply resolve it. Close is a special state: In a normal commercial environment someone would need to confirm the bugfix and close it afterwards. Since we do not have the man power to have such a complete QA cycle gray and green are both considered as resolved. If one of the developer believes that the bugfix is wrong he can simply reopen it.



Editing Bug


Update Bug Simple

Here you can update various bug fields. The Category, Severity, and Reproducibility fields are editable but shouldn't be unless there is a gross miscategorization. Also modifiable are the Assigned To, Priority, Projection, ETA, Resolution, and Duplicate ID fields.

Update bug advanced

Projection and ETA are the only fields that are only modifiable by the Update Bug Advanced page.

Assign To Me

If you are a developer you can use this as a one click assign to 'me'. This is for convenience and for saving time.

Resolve Bug

This option on the View Bugs page allows you to resolve the bug. It will lead you to a page where you can set the resolution state and a duplicate id (if applicable). After choosing that the user can choose to enter a bugnote detailing the reason for the closure. The bug is then set to the Resolved state.

Delete Bug

This option on the View Bugs page allows you to delete an existing bug. This should only be used on frivolous or test bugs. A confirmation screen will prompt you if you really want to delete the bug. Please never delete a bug unless it contains spam or other nonsense. Just the fact that a bug entry is made up on wrong assumptes doesn't make it reasonable to delete it.
